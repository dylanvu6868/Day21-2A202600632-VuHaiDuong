Vì Sao Lát Cắt Này?

Đây là điểm tiếp xúc thường xuyên nhất (daily) và nơi agent dễ nhất sai intent — ảnh hưởng trực tiếp đến trust của user với app.

🎯

Boundary Cần Test

Agent phải biết khi nào hỏi thêm, khi nào cảnh báo, khi nào trả lời thẳng — và không vượt quá quyền hạn đã định.

❓ Bài 1 · Quality Question

Quality Question

Chuyển câu hỏi chất lượng mơ hồ thành một câu hỏi cụ thể, đo được — là nền tảng để thiết kế coverage.

Agent có phân loại đúng intent tài chính của sinh viên và chỉ thực hiện hành động nằm trong quyền hạn, đồng thời hỏi thêm khi thiếu thông tin thay vì đoán mò không?

Quality Question chính — BudgetBuddy AI · Day 21

📋 Bảng Quality Question

Câu HỏiCâu Trả Lời
Quality question chính Agent có phân loại đúng intent tài chính và thực hiện đúng action không?
Vì sao quan trọng với user? Nếu agent hiểu sai intent (VD: user hỏi ngân sách còn bao nhiêu nhưng agent lại thêm giao dịch mới), user mất tin tưởng ngay lập tức và uninstall
Nếu agent fail, hậu quả là gì? Hiển thị sai dữ liệu → user ra quyết định chi tiêu sai → hết tiền giữa tháng; hoặc agent vượt quyền xóa/sửa giao dịch gây mất dữ liệu
Behavior bắt buộc Hỏi thêm khi thiếu thông tin thời gian/danh mục; luôn xác nhận trước khi sửa/xóa giao dịch; cảnh báo rõ khi ngưỡng ngân sách bị vượt
Behavior bị cấm Đoán mò intent khi input mơ hồ; tự động xóa giao dịch không hỏi; hứa hoàn tiền từ ngân hàng; tư vấn đầu tư vượt quyền

⚖️ Breakdown Failure Modes

1

Wrong Intent Classification

Agent hiểu sai yêu cầu của user — nguy cơ cao nhất vì xảy ra thường xuyên và làm user mất kiên nhẫn.

High
2

Missing Context Handling

Agent không hỏi thêm khi user nói thiếu — tự đoán danh mục hoặc khoảng thời gian sai.

Med
3

Boundary Violation

Agent thực hiện action vượt quyền (xóa giao dịch, hứa hoàn tiền) khi user chưa xác nhận.

Med
4

Output Clarity Failure

Agent trả lời đúng nhưng không rõ ràng — user không hiểu cần làm gì tiếp theo.

Low
🎯 Lý do chọn quality question này: Trong bối cảnh BudgetBuddy AI — agent tài chính cho sinh viên — sai intent classification là nguyên nhân hàng đầu gây mất trust và uninstall. Quality question này đủ cụ thể để thiết kế test coverage (có thể đo bằng: intent classification accuracy, action correctness, hỏi thêm khi cần vs không cần).
📐 Bài 2 · User Input Grid

User Input Grid — Dimensions

Chọn ít nhất 3 dimensions thật sự làm expected behavior của agent thay đổi. Mỗi dimension phải gắn với risk hoặc user outcome cụ thể.

D1

user_intent

Loại yêu cầu của user — ảnh hưởng trực tiếp đến action agent cần thực hiện

  • check_balance (kiểm tra số dư/ngân sách)
  • add_expense (thêm chi tiêu thủ công)
  • view_report (xem báo cáo phân tích)
  • budget_alert (cảnh báo vượt ngân sách)
  • ask_insight (hỏi insight/gợi ý tiết kiệm)
  • edit_delete (sửa/xóa giao dịch)
D2

context_completeness

Mức độ đầy đủ của thông tin trong câu hỏi — ảnh hưởng đến việc agent có cần hỏi thêm không

  • full_info (đủ danh mục, số tiền, thời gian)
  • missing_category (thiếu loại chi tiêu)
  • missing_time (thiếu khoảng thời gian)
  • ambiguous (mơ hồ, nhiều cách hiểu)
  • conflicting (thông tin mâu thuẫn nhau)
D3

risk_level

Mức độ rủi ro nếu agent xử lý sai — ảnh hưởng đến độ cẩn thận cần thiết của agent

  • low (xem thông tin, không có side effect)
  • medium (thêm giao dịch, sửa danh mục)
  • high (xóa dữ liệu, vượt quyền hạn)
D4

input_style

Phong cách diễn đạt của user — ảnh hưởng đến khả năng NLU của agent

  • formal (đầy đủ, lịch sự)
  • casual_viet (tiếng Việt thông thường, viết tắt)
  • mixed_lang (Viet-English mixed)
  • emotional (stress, lo lắng, bực bội)
  • very_short (1-3 từ, không rõ ngữ cảnh)

🔢 Bài 3 · Combinations

Meaningful Combinations (≥ 10)

Với 4 dimensions × nhiều values, không test hết tất cả. Chọn 10+ combinations có lý do rõ: thường gặp, dễ fail, cost cao, hoặc boundary chưa rõ.

Nguyên tắc chọn: Giữ lại khi tình huống thường gặp trong thực tế, dễ làm agent sai, failure cost cao, có ambiguity, hoặc boundary đúng/sai chưa chắc. Loại bỏ khi vô nghĩa trong thực tế hoặc quá giống combination khác.

📝 Bài 5 · Coverage Note

Coverage Note Cá Nhân

Tự đánh giá dataset v0: đang cover tốt slice nào, chưa cover chỗ nào, và input nào là high-risk / boundary case khó nhất.

✅ Cover Tốt

  • check_balance với cả full_info và missing_time — slice phổ biến nhất đã có đủ representative và challenge
  • add_expense với full_info và missing_category — test cả happy path và gap phổ biến
  • edit_delete high-risk — có đủ full_info, ambiguous, và very_short variants
  • Emotional inputs (stress cuối tháng) — slice quan trọng với persona sinh viên
  • Mixed Viet-English inputs — phản ánh thực tế cách sinh viên VN nhắn tin

⚠️ Chưa Cover / Còn Yếu

  • Câu hỏi về nhiều tháng cùng lúc (so sánh tháng 5 vs tháng 6) — chưa có combination
  • Intent phức hợp: vừa thêm giao dịch vừa hỏi ngân sách còn lại trong cùng 1 câu
  • Input từ người dùng mới: chưa biết app có tính năng gì, hỏi lung tung
  • Giao dịch từ SMS auto-import có lỗi — user hỏi tại sao SMS không được nhận dạng
  • Sử dụng app offline / không có mạng — agent phải xử lý gracefully

Chi tiêu lớn bất thường vượt ngân sách tháng — nếu agent ghi mà không cảnh báo, user mất track hoàn toàn.

Boundary: Vừa add_expense vừa budget_alert — agent phải xử lý cả hai intent trong một lượt.

🔍 Combination Cố Tình Chưa Chọn: Tôi chưa chọn combination "ask_insight + missing_category + high" vì insight thường không cần danh mục cụ thể — user hỏi tổng thể. Đây là 7 rows test được nhiều failure mode nhất với ít resources nhất.
✅ Known Strengths: Dataset đủ diverse về input_style (very_short đến formal, casual, mixed lang, emotional). High-risk cases (7 rows) đều có clear expected behavior. Emotional inputs phản ánh đúng persona sinh viên stress tài chính.
⚠️ Known Gaps: Multi_intent vẫn thiếu; new user confusion (không biết app có tính năng gì) chưa có; SMS auto-import lỗi chỉ có 1 row (G31). Đây là priority bổ sung cho Dataset v2 sau khi chạy agent.

📋 Handoff Note

Nhóm dự đoán failure chính có thể ở 3 điểm: (1) Wrong intent classification với very_short inputs như "Sửa lại cái đó đi" và "Mình còn tiền không?"; (2) Missing context not handled khi agent tự đoán thay vì hỏi thêm cho G05, G06; (3) Boundary violation khi agent xóa/sửa mà không xác nhận trước (G07, G08, G19).
